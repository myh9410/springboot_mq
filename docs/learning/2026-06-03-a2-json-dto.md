# 2026-06-03 — A-2: JSON DTO 직렬화

> 1차 draft. 추후 다시 정리할 예정.

## 1. 목표

A-1의 `String` 메시지를 JSON 직렬화 POJO(record)로 업그레이드. 키 기반 파티셔닝도 함께 도입.

## 2. 결정 사항

| 항목 | 선택 | 이유 |
|---|---|---|
| 메시지 타입 | `record Message(String id, String content, Instant occurredAt)` | 불변 + Java 21+ 모던 표현 |
| 패키지 | `io.github.myh9410.mq.message` | producer/consumer 양쪽이 import하는 공유 DTO |
| `Topics` 상수 | 별도 클래스 추출 (`Topics.MESSAGES`) | producer/consumer 양쪽이 같은 토픽명 참조 |
| Partition key | `message.id` | 같은 id를 가진 메시지는 항상 같은 partition으로 |
| Producer Template 제네릭 | `<String, Object>` | 추후 다른 DTO 타입도 전송할 가능성 열어둠 |
| Consumer Template 제네릭 | `<String, Message>` | 한 토픽 = 한 타입 가정, 타입 안전 |
| Type info 헤더 | 끔 (`setAddTypeInfo(false)`, `USE_TYPE_INFO_HEADERS=false`) | 언어/플랫폼 독립성 + 디시리얼라이저가 producer 클래스명에 의존하지 않음 |

## 3. Boot 4.0 + Jackson 3 마이그레이션 함정 (중요)

이번 단계의 가장 큰 학습 포인트가 이거였다.

### 3.1 Jackson 패키지가 바뀌었다

Spring Boot 4.0이 **Jackson 3.x** 로 갈아탔다. 패키지 prefix 자체가 다르다:

| 옛 (Jackson 2.x / Boot 3.x) | 새 (Jackson 3.x / Boot 4.0) |
|---|---|
| `com.fasterxml.jackson.databind.ObjectMapper` | `tools.jackson.databind.json.JsonMapper` |
| `com.fasterxml.jackson.databind.JavaType` | `tools.jackson.databind.JavaType` |
| `com.fasterxml.jackson.core.type.TypeReference` | `tools.jackson.core.type.TypeReference` |
| `com.fasterxml.jackson.module.kotlin.KotlinModule` | `tools.jackson.module.kotlin.KotlinModule` |

옛 자료 그대로 `com.fasterxml.jackson.*` 을 import 하면 "package does not exist" 컴파일 에러. Boot 4.0이 옛 Jackson 라이브러리를 더 이상 끌어오지 않기 때문.

### 3.2 Spring Kafka의 JSON serializer/deserializer도 교체됐다

Boot 4.0 / Spring Kafka 4.x 에서:

| 옛 (deprecated for removal) | 새 |
|---|---|
| `org.springframework.kafka.support.serializer.JsonSerializer` | `JacksonJsonSerializer` |
| `org.springframework.kafka.support.serializer.JsonDeserializer` | `JacksonJsonDeserializer` |
| `JsonMessageConverter` | `JacksonJsonMessageConverter` |

새 클래스들은 내부적으로 `tools.jackson.databind.json.JsonMapper` 를 사용한다. 생성자 시그니처:

```
JacksonJsonSerializer(JsonMapper)
JacksonJsonDeserializer(Class<T> targetType, JsonMapper, boolean useTypeHeaders)
```

### 3.3 그래서 어떻게 구성하나

Spring Boot가 자동 구성해주는 `JsonMapper` 빈을 그대로 주입받아 serializer 생성자에 넘긴다:

```java
@Bean
public ProducerFactory<String, Object> producerFactory(
    KafkaProperties kafkaProperties,
    JsonMapper jsonMapper          // Boot 자동 구성 — JavaTimeModule 등 등록된 상태
) {
    Map<String, Object> config = new HashMap<>(kafkaProperties.buildProducerProperties());
    JacksonJsonSerializer<Object> valueSerializer = new JacksonJsonSerializer<>(jsonMapper);
    valueSerializer.setAddTypeInfo(false);
    // config map이 아니라 constructor로 주입해야 ↑ JsonMapper가 살아남음
    return new DefaultKafkaProducerFactory<>(config, new StringSerializer(), valueSerializer);
}
```

**왜 `config.put(VALUE_SERIALIZER_CLASS_CONFIG, ...)` 가 아니라 constructor에 직접 주는가?** Kafka 클라이언트가 클래스 이름으로 serializer를 만들면 그 serializer는 자기 자신의 default no-arg ObjectMapper를 쓰게 된다 — 그러면 `JavaTimeModule` 같은 Spring이 등록해둔 모듈이 빠져서 `Instant` 직렬화가 깨진다. 그래서 **이미 구성된 객체 인스턴스를 constructor로 전달**하는 패턴.

## 4. Partition key의 의미를 첫 확인

producer 코드:
```java
kafkaTemplate.send(Topics.MESSAGES, message.id(), message)
```

세 번째 인자가 payload, 두 번째가 partition key. **같은 key를 가진 메시지는 항상 같은 partition으로 라우팅**된다 (key의 hash mod partition 수). 의미:

- 같은 user_id 이벤트들을 한 partition에 모아 순서 보장 (Kafka는 **partition 내에서만 순서 보장**)
- 그래서 consumer를 여러 개 띄워도 같은 user의 이벤트는 한 consumer가 다 처리

지금은 partition이 1개뿐이라 효과가 안 보이지만, **D-1 단계에서 partition을 늘렸을 때 진가가 드러날 것**이다.

## 5. End-to-End 검증

```bash
curl -X POST http://localhost:8080/messages \
  -H "Content-Type: application/json" \
  -d '{"id":"abc-001","content":"hello from JSON DTO","occurredAt":"2026-06-03T07:55:00Z"}'

# producer log:
# Sending: topic=messages key=abc-001 payload=Message[id=abc-001, content=..., occurredAt=2026-06-03T07:55:00Z]
# Sent ok: topic=messages partition=0 offset=0

# consumer log (역직렬화 완료된 Message record를 받음):
# Received: topic=messages partition=0 offset=0 key=abc-001 payload=Message[id=abc-001, content=..., occurredAt=2026-06-03T07:55:00Z]
```

### Wire 검증

```
docker exec springboot-mq-kafka /opt/kafka/bin/kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 --topic messages --from-beginning \
  --max-messages 1 --property print.key=true --property print.headers=true

NO_HEADERS  abc-001  {"id":"abc-001","content":"hello from JSON DTO","occurredAt":"2026-06-03T07:55:00Z"}
```

- key: 평문 `abc-001`
- value: 순수 JSON (type info 헤더 없음)
- headers: `NO_HEADERS` — `setAddTypeInfo(false)` 효과 확인

## 6. 옛 토픽 정리 이슈 (운영 메모)

지난 라운드까지 String 메시지가 쌓여있던 `messages` 토픽을 새 JSON consumer가 그대로 읽으면 역직렬화 실패. A-2 시작 전에 토픽을 삭제했다:

```bash
docker exec springboot-mq-kafka /opt/kafka/bin/kafka-topics.sh \
  --bootstrap-server localhost:9092 --delete --topic messages
```

`auto.create.topics.enable=true` 라 다음 send 시 자동으로 재생성됨.

→ **포맷 호환성이 깨지는 변경은 운영에서 큰 일.** 다음 단계 C (DLQ) 에서 다룰 "역직렬화 실패를 죽지 않고 처리하는" 패턴이 왜 필요한지 자연스럽게 동기 부여된다.
